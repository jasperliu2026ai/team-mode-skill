# team-mode-skill

把普通 Agent 改造成一套可执行、可协作、可持续推进的团队工作模式。

这个仓库沉淀的是一套已经在真实任务中反复打磨过的团队协作结构：
- `[阿总]`：主控 / 拆解 / 调度 / 对用户负责 / 兜底接管
- `[阿查]`：检索 / 查资料 / 读文档 / 对比方案
- `[阿干]`：执行 / 改文件 / 跑工具 / 交付结果
- `[阿审]`：评审 / 验收 / 风险把关 / 复盘

同时内置两条经过明确约束的能力路线：
- OCR 默认走 `paddleocr-text-recognition`
- 搜索默认走 `search-middle-platform`

## 这玩意解决什么

不是做一个“会聊天的 AI 人设”。
而是把 Agent 变成一个更像小团队的执行单元：
- 会拆分角色
- 会内部协作
- 会持续汇报进度
- 会在用户放权后继续推进
- 会把“真正交付到用户手里”当成完成标准

## 仓库内容

- `team-mode-skill/`：技能本体
- `team-mode-skill.skill`：已打包好的可分发 skill 文件

## 核心约束

### 1. 固定角色模型

默认使用：
- `[阿总]`
- `[阿查]`
- `[阿干]`
- `[阿审]`

### 2. 阿总必须兜底

任何角色卡住、缺席、慢响应，`[阿总]` 必须马上接管，不能让用户干等。

### 3. 用户放权后，默认持续推进

不要把内部障碍一股脑抛回给用户。
能内部解决就内部解决。
除非涉及破坏性操作、外发、隐私、权限边界，否则默认继续推进。

### 4. OCR 路由固定

OCR / 图片识别 / 截图文字提取场景，优先走：
- `paddleocr-text-recognition`

### 5. 搜索路由固定

多源检索 / 技术检索 / 图片聚合 / 答案收敛场景，优先走：
- `search-middle-platform`

### 6. 验收标准是真交付

本地生成文件，不等于用户收到。
流程跑通，不等于任务完成。
真正的完成标准是：
- 用户拿到结果
- 或者拿到证据充分的 blocker

## 使用方式

### 方式一：直接参考仓库内容

把 `team-mode-skill/` 目录放进你的 Agent skills 目录，然后让 Agent 读取并启用它。

### 方式二：导入打包文件

使用仓库根目录下的：
- `team-mode-skill.skill`

把这个 skill 导入到支持 AgentSkills 的环境中。

## 适用场景

- 想把单 Agent 改成“团队协作模式”
- 想复制一套稳定的多角色分工结构
- 想让 Agent 默认更主动、更会推进、更会交付
- 想把 OCR / 搜索 / 评审这些能力纳入统一工作流

## 注意

这是一个“工作方式 skill”，不是某个单点 API 封装。
它更像一个团队执行操作系统。
你拿到仓库后，可以继续把自己的内部规范、工具路线、交付标准追加进去。
